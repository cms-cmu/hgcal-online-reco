# Bug Report: `encode_wafers.py` — Zero Latent Space Outputs

## Summary

Four bugs were identified in `encode_wafers.py`. Together they prevented the script from
running at all (SyntaxError) and would have produced malformed condition tensors even if
it had run, leading to incorrect (near-zero) encoder outputs.

---

## Bugs Fixed

### Bug 1 — `continue` outside a loop (SyntaxError) · Line 260

**Root cause of "all zeros"**

```python
# BEFORE (broken)
except Exception as e:
    print(f"    ERROR: {e}")
    import traceback
    traceback.print_exc()
    continue          # ← SyntaxError: 'continue' not inside a loop
```

`load_root_file` processes a single file; there is no enclosing loop. Python raises a
`SyntaxError` at parse time, meaning the entire module fails to import and **no encoding
could take place at all**. Any existing `.npz` files containing zeros were generated by
an earlier version of the script.

**Fix:** Changed `continue` → `raise` so that errors in the file-loading step propagate
to the caller (`process_single_file`), which already has its own `try/except` that logs
the failure and moves on.

---

### Bug 2 — `file_index` undefined (NameError) · Line 205

```python
# BEFORE (broken)
file_indices.extend([file_index] * n_wafers)   # file_index never defined

# AFTER
file_indices.extend([0] * n_wafers)            # single-file function → always 0
```

`load_root_file` was refactored from a version that iterated over a list of files but
the loop variable `file_index` was never replaced with a concrete value. Would have
raised a `NameError` at runtime.

---

### Bug 3 — `files` undefined (NameError) · Line 275

```python
# BEFORE (broken)
print(f"  Total files: {len(files)}")     # 'files' never defined in this scope

# AFTER
print(f"  Total files: 1")               # the function processes exactly one file
```

Same refactoring artifact as Bug 2. `files` was the list of ROOT paths in the original
multi-file loop; it no longer exists in the single-file function.

---

### Bug 4 — `one_hot_wafertype` width depends on runtime data · Line 216

**Root cause of near-zero outputs in pre-existing `.npz` files**

```python
# BEFORE (broken)
one_hot_wafertype = np.eye(np.max(wafer_type) + 1)[wafer_type]
```

The number of one-hot columns is computed dynamically from the highest wafer-type value
seen in a given file. The encoder model expects a fixed condition vector of shape `(8,)`:

| Feature | Width |
|---|---|
| eta | 1 |
| wafer_v | 1 |
| wafer_u | 1 |
| one_hot_wafertype | **3** (types 0, 1, 2) |
| sum_CALQ | 1 |
| layers_normalized | 1 |
| **Total** | **8** |

If a file only contains wafer types 0 and 1, `np.max = 1` → `np.eye(2)` → 7-column
condition vector, causing a shape mismatch with the model input layer and producing
garbage/zero outputs silently.

```python
# AFTER
one_hot_wafertype = np.eye(3)[wafer_type]   # hardcoded to 3 types, matching condition_shape=(8,)
```

## Follow-Up: Plots Still Showed "All Zeros" After Encode Fix

### Diagnosis

After the above fixes, `encode_wafers.py` was re-run and produced valid `.npz` files.
However, `plot_latent_images.png` still appeared entirely zero (uniform dark purple).

An NPZ diagnostic revealed the latent space **is not all zeros** — the encoder is
working correctly:

| Metric | Value |
|---|---|
| Total wafers | 5,464,973 |
| Wafers with non-zero latent | 32,554 (0.6%) |
| Latent value range | [−0.80, 0.75] |
| First non-zero wafer index | **49** |
| Non-zero elements | 241,630 / 87,439,568 |

Per-eLink breakdown (all producing non-zero output):

| eLink | Wafers | Range | Non-zero elements |
|---|---|---|---|
| 2 | 2,769,843 | [−0.50, 0.75] | 67,497 |
| 3 | 300,751 | [−0.19, 0.13] | 14,476 |
| 4 | 780,043 | [−0.45, 0.64] | 43,828 |
| 5 | 1,614,336 | [−0.80, 0.69] | 115,829 |

The high sparsity (~99.7% zeros) is expected from the heavy fixed-point quantization
in the encoder. Most wafers carry low-energy deposits that quantize to zero.

### Root Cause

`plot_latent` in `plot_data.py` plotted **wafers 0–15** (the first 16), which all
happen to be zero. The first non-zero wafer is at index 49, so the plot showed uniform
dark purple and appeared "all zeros."

## Bug 5 — `plot_latent` shows wrong wafers · `plot_data.py` line 152

### Changes to `plot_data.py`

**Latent image selection** (was: first 16, now: top 16 by L1 norm):

```python
# BEFORE (broken — shows first 16 wafers, which are all zero)
num_images = 16
fig, axes = plt.subplots(4, 4, figsize=(12, 12))
fig.suptitle('L1 Latent Space (4x4 representations of first 16 wafers)')
for i, ax in enumerate(axes.flatten()):
    if i < len(latent):
        img = latent[i].reshape(4, 4)
        ax.imshow(img, cmap='viridis', aspect='auto')
        ax.set_title(f'Wafer {i}')

# AFTER — selects 16 most active wafers by L1 norm
l1_norms = np.sum(np.abs(latent), axis=1)
top_indices = np.argsort(l1_norms)[-16:][::-1]  # top 16 by activity
vmax = np.max(np.abs(latent[top_indices]))
for i, ax in enumerate(axes.flatten()):
    idx = top_indices[i]
    img = latent[idx].reshape(4, 4)
    im = ax.imshow(img, cmap='RdBu_r', aspect='auto', vmin=-vmax, vmax=vmax)
    ax.set_title(f'Wafer {idx}\n(L1={l1_norms[idx]:.2f})')
```

**Distribution plot** (was: single panel dominated by zero spike, now: two panels):

- Left panel: all values including zeros (log scale)
- Right panel: non-zero values only, showing the actual distribution shape

**New: latent heatmap** (`plot_latent_heatmap.png`):

- Samples 200 random non-zero wafers
- Displays 16×200 heatmap of latent dimension activations using `RdBu_r`
- Shows which latent dimensions are most active across the dataset
